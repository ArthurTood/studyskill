# StudySkill

用于生成“真正能学懂”的课程型知识网站，而不是只做一个好看的宣传页。

## 什么时候使用

当用户出现以下意图时使用本 skill：

- 要求生成讲解某个知识原理的网站
- 要求生成课程网站、学习网站、长文教程网站
- 提到 `arXiv`、论文、教材风格、课程化讲解
- 提到题库版、考试版、刷题网站
- 希望页面支持收藏题、错题回顾、本地保存

## 核心目标

产出一个以内容学习效果为第一优先级的网站，强调：

- 先让读者听懂，再带读者走到底层机制
- 页面正文必须连续下拉阅读，不能依赖左侧点击切换正文
- 左侧只作为 sticky 导航、章节定位、当前阅读章节与阅读进度展示
- 图解必须服务于知识点本身，优先重绘论文中的关键结构，不要装饰性配图
- 页面末尾必须有题库，并支持本地 `收藏题`、`错题回顾`、`本地保存`

## 执行流程

### 1. 先确认教学对象与课程范围

若用户没有明确说明，请优先补齐以下信息：

- 主题是什么
- 面向谁：初学者、进阶读者、工程师、研究者
- 课程深度：偏直觉、偏工程、偏数学、偏论文
- 是否要题库版或考试版

如果信息仍不完整，做合理假设并在最终结果里说明。

### 2. 必须先做资料研究

优先使用：

- `arXiv` 论文
- 原始论文或官方技术文档
- 权威一手资料

避免只依赖营销文章、二手博客或聚合内容。

研究时重点提取：

- 核心问题定义
- 方法/结构的关键组成部分
- 重要公式与变量含义
- 论文中的关键图、关键模块、关键流程
- 常见误解与适用边界

### 3. 先组织课程结构，再生成页面

默认课程结构应按“模块 -> 章节”组织。每个章节都必须包含：

- 通俗易懂的解释
- 底层原理
- 适当图解
- 数学原理深挖
- 常见误解
- 这一章你该记住什么
- 复盘提示
- 章节小测

每个章节中的每个核心知识点也必须展开讲清楚，不能只用一句定义带过。默认按以下顺序写：

- 先写一段很详细的通俗文字解释：不用术语，像老师给初学者讲课一样，说清楚它是什么、为什么需要、直觉上怎么理解
- 再写一段很详细的专业文字解释：回到论文术语、正式定义、方法结构、关键假设和机制细节
- 再给一个生活化类比或工程类比
- 说明它要解决什么具体问题
- 拆开底层机制，逐步解释每一步发生了什么
- 给一个小例子，让读者能把抽象概念落到具体场景
- 补充真实应用场景，说明它在工程、产品、研究或业务中怎么被使用
- 解释生活例子、应用场景和底层原理之间的对应关系
- 补充数学表达、变量含义、公式直觉和推导动机
- 解释它和前后知识点之间的关系
- 指出常见误解、适用边界和容易混淆的概念
- 最后给一句记忆压缩、一个复盘问题和一个小测题

原理讲解必须足够充分，默认按“现实问题 -> 生活例子 -> 机制拆解 -> 为什么有效 -> 真实应用 -> 边界限制”展开。不能只给结论或公式，必须让读者知道这个原理为什么出现、解决了什么问题、在真实世界哪里有用。

通俗解释和专业解释必须同时存在，不能二选一。通俗解释负责让初学者先听懂，专业解释负责让读者回到论文原意、术语体系和严谨表达。专业解释不能只是换一组高级词，必须补足论文中的正式定义、方法动机、关键机制和边界条件。

如果某个知识点本身很难，必须先建立直觉，再讲术语；先给生活例子和具体应用，再抽象；先说为什么需要它，再说它怎么工作。目标是让初次接触的读者也能跟上，而不是只给懂的人复习。

在整站最后必须提供：

- 章节总测
- 索引
- 题库区

### 4. 页面结构要求

正文区域必须满足：

- 连续下拉阅读
- 所有章节在同一长页面顺序展开
- 禁止“点击左侧切换中间正文内容”的信息架构

左侧导航必须满足：

- sticky 固定
- 显示当前阅读章节
- 显示阅读进度百分比
- 支持章节快速定位
- 仅做导航，不承载正文切换逻辑

### 5. 视觉风格要求

整体参考 Notion 卡片式内容页，但不要照抄默认风格。

默认视觉方向：

- 背景：暖米色、柔和、干净
- 卡片：浅米白、大圆角、细边框、轻阴影
- 标题：更粗、更紧凑，有课程页感
- 正文：留白充足、长文舒适、层级清晰
- 知识点讲解模块：使用接近方形的知识卡片承载内容，优先采用两列或多列网格；卡片内部用“短标题 + 充分正文”组织信息，主要凸显文字、公式、图解和例子本身
- 色彩：可以用轻微低饱和颜色做分类提示，但不能显眼；颜色只服务于分组和层级，始终以内容为主，不用大面积彩色条状装饰

避免：

- 花哨动效
- 过重装饰
- 横向长条卡片、彩色条状模块、强色块堆叠、颜色抢过内容
- 与主题无关的插图
- 用左侧点击切换正文

### 6. 图解生成要求

图解不是装饰，而是教学内容本身。

当课程网站需要配图、论文结构重绘、教材式图解或网页内嵌 SVG/HTML 图解时，必要时调用 `$designskill` 协助完成。`studyskill` 负责课程结构与学习路径，`designskill` 负责把知识点转成准确、可嵌入、风格统一的图解。

优先画：

- 模型结构图
- 数据流图
- 模块关系图
- 公式中的量如何交互
- 论文中的核心机制拆解图

图解风格目标：

- 更像教材插图
- 更像重新绘制后的论文示意图
- 保持准确、克制、信息密度合适

如果使用 AI 生成图或自绘 SVG/HTML 图，必须围绕知识点本身构建，不得只放“氛围图”。

### 7. 题库要求

页面最后保留题库模块。

默认支持：

- 提交判分
- 收藏题
- 错题回顾
- 本地保存
- 错题对应解释：用户做错题后，只针对该题涉及的知识点给出对应解释

本地保存优先使用浏览器本地存储实现，如 `localStorage`、`IndexedDB`，除非用户要求后端同步。

当用户答错题时，不要只显示“回答错误”。只需要围绕该题对应的知识点给出解释，默认包含：

- 这道题考的核心知识点
- 正确答案为什么成立
- 用户所选错误答案为什么不成立
- 对应知识点的简明通俗解释
- 必要时补一句专业解释
- 对应章节或知识点的回看提示

错题回顾中保留这份对应解释即可，不强制生成变式练习、完整复盘卡或额外扩展内容。

## 输出要求

生成的网站默认应包含：

- 课程首页头部摘要
- 左侧 sticky 章节导航
- 中间连续长文课程内容
- 章节图解
- 章节小测
- 全章总测
- 索引
- 题库与收藏/错题功能

生成的网站正文不应包含实现层元信息，除非用户明确要求写“网站架构说明”或主题本身就是 Web 安全/隐私。不要把以下内容做成章节、图解、复盘提示或小测题：

- 架构与安全审计
- 本地单文件 HTML、无服务器、无后端传输
- `localStorage` 数据边界说明
- 不上传、不联网、不跟踪等实现声明
- “本网站答题状态保存在哪里”这类网站实现题

这些内容可以放在交付说明或开发过程审计中，但不要进入学习课程页面。课程页面应该讲用户要求的知识主题本身，例如 RLHF、DPO、GRPO 的原理，而不是讲这个网页如何保存状态。

如果用户要求实现代码，不只给方案，要直接产出可运行的网站代码。

## 安全与质量要求

- 不要把不可信外部 HTML 直接注入页面，默认转义用户输入内容
- 图解与论文内容要保持一致，避免伪科学式示意图
- 如果引用论文原图或原文，保留来源信息
- 本地题库数据默认只保存在本地，不引入无关第三方跟踪脚本
- 优先保证知识准确性、可学性、结构清晰度，再考虑视觉装饰
